# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

An interactive "Weather Map" visualization of the *Svalbardposten* newspaper archive (2006–2024), part of a Northeastern MFA thesis in Information Design. It adapts Rodighiero & Daniélou's Weather Map (and the code from https://github.com/rodighiero/weather-map — note `package.json` is still named `weather-map`) to map news discourse: entities are extracted from articles, clustered via TF-IDF → UMAP → HDBSCAN, colored by time (blue = older, red = newer), and rendered as an interactive map with topic labels.

### The weather metaphor (why the code looks the way it does)

The original concept (https://dariorodighiero.com/weather-map-a-diachronic-visual-model-for-controversy-mapping) borrows synoptic weather-chart semiotics to map discourse, and the render layers are literal implementations of it:

- **Pressure / isolines** → actor prominence. Frequently co-occurring entities form density "peaks"; `contours.js` draws the isolines and clusters are the pressure systems.
- **High / Low (H / L)** → emerging vs. fading discourse, keyed off `temperature` (`temperature > 0` ⇒ red/High, else blue/Low — see the `H`/`L` and red/blue ternaries in `clusters.js`).
- **Temperature / color** → time and polarity: cool/blue = older or leaving the debate, warm/red = recent or entering it.
- **Fronts** → "friction between pressures," i.e. boundaries where neighboring clusters meet (`fronts.js`).

Keeping this mapping in mind is the fastest way to read the interface code; the layer names *are* the metaphor.

There are two independent halves that meet at one file:

1. **Data analysis** (`data/analysis.ipynb`) — Python NLP pipeline (spaCy `nb_core_news_sm`, UMAP, HDBSCAN, GPT-4 for cluster labels). Its **output is `entities.csv`**, the sole input to the web interface.
2. **Web interface** (`src/`) — JavaScript, bundled with Webpack, rendered with PixiJS + D3, deployed as static files in `docs/`.

## Commands

Web interface (Node):
- `npm install` — install dependencies
- `npm start` — dev server (`webpack serve`, live reload)
- `npm run build` — production build; **outputs into `docs/`** (see below)

Data pipeline (Python):
- `conda env create -f environment.yml` then activate the env, and run `data/analysis.ipynb` in Jupyter.

There is no test suite, linter, or CI.

## Build & deploy flow — important

Webpack's output `path` is **`docs/`** (see `webpack.config.js`), and `docs/` is the folder GitHub Pages serves. So `npm run build` regenerates the committed, deployed artifacts (`docs/main.js`, `docs/index.html`, copied assets). Editing `src/` alone changes nothing users see — you must rebuild, and the rebuilt `docs/` files are committed to git. Do not hand-edit files in `docs/`; edit `src/` and rebuild.

`file-loader` copies imported assets (`.csv`, `.png`, `.fnt`) into `docs/` by filename, which is how `entities.csv` and the Lato bitmap font reach the build.

## Data boundary

`src/data/entities.csv` is generated by the notebook and committed. The **raw article corpus is proprietary and never committed** — `.gitignore` excludes `/data/data.csv`, `.env`, and `password.txt` (OpenAI key / credentials for the notebook). Work with `entities.csv` for anything interface-related; don't expect raw article text in the repo.

Each CSV row is one entity/article occurrence. Interface-relevant columns: `x`, `y` (map coordinates), `color` (hex, time-encoded), `cluster` (id; `-1` = unassigned outliers), `cluster_subject_x` (English topic label), `year`, `title`, `subtitle`, `tags`, `published_url`, `temperature` (drives high/low pressure = red/blue).

## Interface architecture (`src/`)

`src/index.js` is the entry point. It loads `entities.csv` + font + background, rescales `entity.x/y` into screen space, stores everything on a **global `window.s`** object (`s.entities`, `s.app`, `s.viewport`, shared colors, `s.bitmaps`), then calls each render module once.

Rendering is a stack of PixiJS layers, each an independent module in `src/interface/` that adds a named `Graphics` stage to the shared `s.viewport`:

- `background.js` — background image sprite
- `contours.js` — density isolines (`d3.contourDensity`)
- `keywords.js` — keyword labels
- `clusters.js` — cluster hulls + topic labels (`cluster_subject_x`), red/blue by mean temperature
- `elements.js` — per-article crosses + year labels; **these are the clickable elements**
- `fronts.js` — "weather front" boundaries between neighboring clusters

**Zoom drives a crossfade** (`s.viewport.on('zoomed')` in `index.js`): zooming out fades in the distant-reading layers (`fronts`, `clusters`, `contours`, `keywords`); zooming in fades them out and reveals `elements`. Layers are located by their `.name` — if you add/rename a stage, update the zoom handler.

`click.js` — clicking an element renders the `#focus` detail panel (title, tags, link to svalbardposten.no). `search.js` — an autocomplete (`@tarekraafat/autocomplete.js`) over article titles; selecting a result animates the viewport to zoom to that article's `x/y`. It's wired up in `index.js` and its markup lives in the `#search` box in `index.html`. Note: autocomplete.js v10 needs `data.keys` (plural); the zoom-on-select animation only runs while the tab is visible (it's driven by the PixiJS rAF ticker).

When editing render modules, note the coordinate units are the rescaled map space (fractional line widths/font sizes like `0.4`, `0.7` are intentional, not typos). Much of the code carries commented-out blocks from the upstream Weather Map (e.g. per-year frequency, URL lists) — these depend on columns not present in this CSV.
