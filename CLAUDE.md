# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

An interactive "Weather Map" visualization of the *Svalbardposten* newspaper archive (2006–2024), part of a Northeastern MFA thesis in Information Design. It adapts Rodighiero & Daniélou's Weather Map (and the code from https://github.com/rodighiero/weather-map — note `package.json` is still named `weather-map`) to map news discourse: entities are extracted from articles, clustered via TF-IDF → UMAP → HDBSCAN, colored by time (blue = older, red = newer), and rendered as an interactive map with topic labels.

### The weather metaphor (why the code looks the way it does)

The original concept (https://dariorodighiero.com/weather-map-a-diachronic-visual-model-for-controversy-mapping) borrows synoptic weather-chart semiotics to map discourse, and the render layers are literal implementations of it:

- **Pressure / isolines** → actor prominence. Frequently co-occurring entities form density "peaks"; `contours.js` draws the isolines and clusters are the pressure systems.
- **High / Low** → emerging vs. fading discourse, keyed off `temperature` (`temperature > 0` ⇒ red/High, else blue/Low — see the red/blue ternary on the cluster label tint in `clusters.js`).
- **Temperature / color** → time and polarity: cool/blue = older or leaving the debate, warm/red = recent or entering it.
- **Fronts** → "friction between pressures," i.e. boundaries where neighboring clusters meet (`fronts.js`).

Keeping this mapping in mind is the fastest way to read the interface code; the layer names *are* the metaphor.

There are two independent halves that meet at one file:

1. **Data analysis** (`data/analysis.ipynb`) — Python NLP pipeline (spaCy `nb_core_news_sm`, UMAP, HDBSCAN, GPT-4 for cluster labels). Its **output is `entities.csv`**, the sole input to the web interface.
2. **Web interface** (`src/`) — JavaScript, bundled with **Vite**, rendered with **PixiJS 8** + D3, deployed as static files in `docs/`.

## Commands

Web interface (Node):
- `npm install` — install dependencies
- `npm start` (or `npm run dev`) — Vite dev server (default http://localhost:5173, hot reload)
- `npm run build` — Vite production build; **outputs into `docs/`** (see below)
- `npm run preview` — serve the built `docs/` locally

Data pipeline (Python):
- `conda env create -f environment.yml` then activate the env, and run `data/analysis.ipynb` in Jupyter.

There is no test suite, linter, or CI.

## Build & deploy flow — important

Vite's `build.outDir` is **`docs/`** (see `vite.config.js`), and `docs/` is the folder GitHub Pages serves. So `npm run build` regenerates the committed, deployed artifacts (`docs/index.html`, `docs/assets/*` hashed JS/CSS). `emptyOutDir` clears `docs/` on each build. Editing `src/` alone changes nothing users see — you must rebuild, and the rebuilt `docs/` files are committed to git. Do not hand-edit files in `docs/`; edit `src/` and rebuild.

`base: './'` makes asset URLs relative so the build works from the Pages subpath (`/svalbard-workshop/`). The HTML entry is **`index.html` at the repo root** (with `<script type="module" src="/src/index.js">`), not in `src/`.

`public/` holds the **runtime static assets** (`Lato.fnt`, `Lato.png`, `entities.csv`) — Vite copies them into `docs/` **verbatim, unhashed**. This matters: `Lato.fnt` references `Lato.png` by a fixed page name, so the font page image must not be bundled/hashed. `index.js` loads these from `import.meta.env.BASE_URL` (e.g. `Assets.load(base + 'Lato.fnt')`), which resolves correctly on the subpath.

## Data boundary

`public/entities.csv` is generated by the notebook and committed. The **raw article corpus is proprietary and never committed** — `.gitignore` excludes `/data/data.csv`, `.env`, and `password.txt` (OpenAI key / credentials for the notebook). Work with `entities.csv` for anything interface-related; don't expect raw article text in the repo.

Each CSV row is one entity/article occurrence. Interface-relevant columns: `x`, `y` (map coordinates), `color` (hex, time-encoded), `cluster` (id; `-1` = unassigned outliers), `cluster_subject_x` (English topic label), `year`, `title`, `subtitle`, `tags`, `published_url`, `temperature` (drives high/low pressure = red/blue).

## Interface architecture (`src/`)

`src/index.js` is the entry point. It loads `entities.csv` + font, rescales `entity.x/y` into screen space, stores everything on a **global `window.s`** object (`s.entities`, `s.app`, `s.viewport`, shared colors), then calls each render module once.

Rendering is a stack of PixiJS layers, each an independent module in `src/interface/` that adds a labelled `Container` stage to the shared `s.viewport`. **PixiJS 8 note:** a `Graphics` may no longer hold children, so stages that carry child bitmaps/graphics are `Container`s (with a separate `Graphics` child for any drawing — see `clusters.js`); a stage that only draws (`contours.js`) stays a `Graphics`.

- `contours.js` — density isolines (`d3.contourDensity`)
- `clusters.js` — cluster hulls + topic labels (`cluster_subject_x`), red/blue by mean temperature
- `elements.js` — per-article crosses + year labels; **these are the clickable elements** (`eventMode`/`on('pointertap')`)
- `fronts.js` — "weather front" boundaries between neighboring clusters

**Zoom drives a crossfade** (`s.viewport.on('zoomed')` in `index.js`): zooming out fades in the distant-reading layers (`fronts`, `clusters`, `contours`); zooming in fades them out and reveals `elements`. Layers are located by their `.label` — if you add/rename a stage, update the zoom handler.

`click.js` — clicking a year label renders the `#focus` detail panel (title, tags, link to svalbardposten.no); exported as a **named** `click` (not a default). The search feature (autocomplete title search that zoomed to a result) was **removed**, along with its `@tarekraafat/autocomplete.js` dependency; restore both from git if needed (last full version at `56c9e8b`).

When editing render modules, note the coordinate units are the rescaled map space (fractional line widths/font sizes like `0.4`, `0.7` are intentional, not typos). Some modules still carry commented-out blocks from the upstream Weather Map (e.g. the front lines in `fronts.js`) that depend on columns not present in this CSV (`slope`, `frequency_norm`, `type`, `name`).
